Stage 3 — MAJCAT applicability check

applicable_majcats + filter_to_applicable

TABLE MASTER_SZ_APPLICABLE

Lookup: WHERE UPPER(APPLICABLE) = 'Y'

Suppose M_TEES_HS is in the applicable list → row survives.

Stages 4–5c — Combination grouping & filters

group_and_count

count = COUNT(rows) per (MAJCAT, GEN_ART, CLR), broadcast per row

Suppose Polo BLK has only 2 sizes (M and L) → count = 2

group_fnl_q_sum

fnl_q_sum = SUM(FNL_Q) per (MAJCAT, GEN_ART, CLR), broadcast per row

Polo BLK: M has FNL_Q=4, L has FNL_Q=6 → fnl_q_sum = 10

filter_count + filter_fnl_q_sum + filter_placeholder

  • count = 2 ≤ 2 ✓ (user threshold)
  • fnl_q_sum = 10 ≤ 50 ✓ (user threshold)
  • CLR = 'BLK' ≠ 'A' ✓, SZ = 'M' ≠ 'A' ✓

Row survives → tagged status = '1sz'

Stages 7a–7b — RDC filter & cross-join

apply_rdc_filter + cross_join

User picked RDC = DH24. Both MSA and stores reduced to DH24 only.

INNER JOIN on RDC. Our Polo BLK row in RDC DH24 joins with TR05 (also DH24).

After cross-join: one row with ST_CD=TR05, ST_NM=HAVERI, RDC=DH24, days=9, MAJCAT=M_TEES_HS, GEN_ART=12345, CLR=BLK, SZ=M, count=2, fnl_q_sum=10

Stage 8 — Lookup cont (size contribution %)

enrich_cont

TABLE Master_CONT_SZ

Join keys: (ST_CD, MAJ_CAT, SZ)

Lookup row: (TR05, M_TEES_HS, M) → CONT = 0.5

(Size M accounts for 50% of category sales at this store.)

cont = 0.5

Stage 9 — DISP_Q + SAL_PD + ALC_D from ARS_CALC_ST_MAJ_CAT

enrich_calc_maj_cat

TABLE ARS_CALC_ST_MAJ_CAT

Join keys: (ST_CD, MAJ_CAT) — maj_cat grain

Lookup row: (TR05, M_TEES_HS)

Stage 9a — Overwrite SAL_PD with option-grain

enrich_sal_pd_option_grain

TABLE MASTER_GEN_ART_SALE

Join keys: (ST_CD, MAJ_CAT, GEN_ART_NUMBER, CLR) — option grain

Lookup: (TR05, M_TEES_HS, 12345, BLK) → SAL_PD = 0.8

(Polo BLK specifically sells 0.8/day at TR05 — slightly higher than category baseline 0.6.)

SAL_PD overwritten: 0.6 → 0.8

Stage 9b — ACS_D from ARS_GRID_MJ

enrich_grid_mj_base

TABLE ARS_GRID_MJ

Join keys: (WERKS=ST_CD, MAJ_CAT)

Lookup: (TR05, M_TEES_HS) → ACS_D = 16 (fixture buffer days)

ACS_D = 16

Stage 9c — Compute var_art_disp

compute_var_art_disp

var_art_disp = ACS_D × cont = 16 × 0.5 = 8

Stage 10 — Compute MBQ_SZ

compute_mbq_sz

MBQ_SZ = (DISP_Q + SAL_PD × days) × cont
= (14 + 0.8 × 9) × 0.5
= (14 + 7.2) × 0.5
= 21.2 × 0.5
= 10.6 → 11 (rounded)

Stage 11 — STK_TTL from ARS_GRID_MJ_VAR_ART

enrich_grid_mj

TABLE ARS_GRID_MJ_VAR_ART

Join keys: (WERKS=ST_CD, MAJ_CAT, SZ) — size grain

SUM aggregated across variants of size M.

Lookup: (TR05, M_TEES_HS, M) → STK_TTL = 3

Stage 12 — Compute REQ

compute_req

REQ = ROUND(MAX(MBQ_SZ − STK_TTL, 0), 0)
= MAX(11 − 3, 0)
= 8

Stage 13 — Compute MAX_DAILY_SALE (age-aware)

enrich_listing — 5 sub-steps

Step a: AUTO_GEN_ART_SALE = SAL_PD

Just an alias: AUTO_GEN_ART_SALE = 0.8

Step b: Load L-7-day sale from ARS_GRID_MJ_GEN_ART

Column L-7 DAYS SALE-Q = 5.6 for this option

L7_DAILY = 5.6 / 7 = 0.8

Step c: Compute PER_OPT_SALE from opt-sale grid

Active grid MJ_STKD has STKD_MBQ=30, STKD_DISP_Q=10 for this option.

PER_OPT_SALE = ((30 − 10) / 10 × 16) / 14
= (2.0 × 16) / 14
= 32 / 14
= 2.29

Step d: Load AGE from MASTER_GEN_ART_AGE

Lookup: (TR05, M_TEES_HS, 12345, BLK) → AGE = 5 days (a new article)

Step e: Compute MAX_DAILY_SALE with AGE switch

AGE = 5 → AGE < 15 → use 3-input MAX

MAX_DAILY_SALE = MAX(L7_DAILY, AUTO_GEN_ART_SALE, PER_OPT_SALE)
= MAX(0.8, 0.8, 2.29)
= 2.29   (PER_OPT_SALE wins)

If AGE were ≥ 15 (old article), we'd exclude PER_OPT_SALE:

[hypothetical OLD] MAX_DAILY_SALE = MAX(0.8, 0.8) = 0.8

Stage 14 — Compute SALE_VAR_ART

compute_sale_var_art

SALE_VAR_ART = MAX_DAILY_SALE × cont
= 2.29 × 0.5
= 1.145

Stage 15 — Compute MBQ_VAR

compute_mbq_var

MBQ_VAR = (MAX_DAILY_SALE × days) + ACS_D
= (2.29 × 9) + 16
= 20.61 + 16
= 36.61
= 37 (rounded)

Stage 16 — Compute VAR_REQ

compute_var_req

VAR_REQ = ROUND(MAX(MBQ_VAR − STK_TTL, 0), 0)
= MAX(37 − 3, 0)
= 34

Stage 17 — Load MSA_REMAIN pool + ST_RANK

enrich_msa_remain_and_rank

Part A — MSA_REMAIN (warehouse pool for this size in this RDC)

TABLE ARS_MSA_TOTAL re-aggregated

MSA_REMAIN = SUM(FNL_Q) across all SLOCs
per (RDC, MAJ_CAT, GEN_ART, CLR, SZ)

For (DH24, M_TEES_HS, 12345, BLK, M) → MSA_REMAIN = 12 units in the warehouse

Part B — ST_RANK

TABLE ARS_STORE_RANKING

Join: (ST_CD, MAJ_CAT) → Lookup: (TR05, M_TEES_HS) → ST_RANK = 3

(TR05 is the 3rd-priority store for this category.)

Reading the Result

One row produced one shipment decision: ship 2 units of Polo BLK size M to store TR05.

  • The system thought TR05 needs 34 units variant-wise (VAR_REQ) and 8 units size-wise (REQ).
  • It would have shipped MIN(34, 8) = 8 — but the warehouse only had 2 left after higher-ranked stores took theirs.
  • So TR05 got 2 — the most it could.
  • Subsequent (lower-ranked) stores in this pool will get 0 (POOL_EMPTY).
