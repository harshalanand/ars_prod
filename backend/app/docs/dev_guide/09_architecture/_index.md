# Database Architecture — Developer Reference

ARS runs on **two SQL Server databases on the same instance** —
splitting data by purpose lets each scale and lock independently.

```
                Azure SQL instance (ars-v2retail-sql)
                            │
          ┌─────────────────┴─────────────────┐
          ▼                                    ▼
    System DB (Claude)                   Data DB (Rep_Data)
    ────────────────────                 ──────────────────
    rbac_users / roles / perms           ET_STORE_STOCK
    rls_categories                       ET_MSA_STK
    audit_log                            ARS_MSA_TOTAL / GEN_ART / VAR_ART
    upload_jobs                          ARS_GRID_*  (one per grid)
    table metadata                       ARS_pend_alc
                                         ARS_ALLOCATION_MASTER + DETAIL
                                         ARS_ALLOCATION_AUDIT
                                         ARS_CONTRIB_PRESETS / MAPPINGS
                                         ARS_CONTRIB_RESULTS_<jobid>
                                         vw_master_product
                                         VW_ET_MSA_STK_WITH_MASTER
                                         Trend_*
```

## Why two databases?

1. **Independent locking** — a 2-minute upload locks parts of Rep_Data;
   the login/audit reads don't have to wait.
2. **Independent backup/restore** — losing business data doesn't lose
   user accounts and audit trail.
3. **Independent permissions** — easier to give a planner read-only
   access to data without exposing user passwords.

## Engine setup — `app/database/session.py`

Two engines, identical config:

```python
system_engine = create_engine(settings.DATABASE_URL, ...)
data_engine   = create_engine(settings.DATA_DATABASE_URL, ...)
```

Both have:

- `pool_size=15, max_overflow=25` — handles ~20 concurrent users
- `pool_recycle=300` — Azure recommends 5-min recycle to avoid stale conns
- `pool_pre_ping=True` — auto-test conn before checkout
- `fast_executemany=True` — pyodbc bulk insert flag, critical for speed
- ODBC string includes `ConnectRetryCount=5;ConnectRetryInterval=10` — auto-recovers from Azure SQL transient 40613 errors

## Critical platform constraint: Azure SQL Database

Both DBs are on Azure SQL DB (PaaS). That means:

- **Recovery model is permanently FULL.** You cannot ALTER it.
- **Microsoft auto-runs log backups** every ~5–10 min.
- **Per-tier log size cap** — exceed it and writes fail with **error 9002**.
- **DBCC SHRINKFILE on system DBs is restricted.**
- **Cross-database queries are limited** (no synonyms / linked-server tricks).

The auto-resolve setting `AUTO_RESOLVE_LOG_BACKUP_WAIT` flips a DB
to SIMPLE recovery if log truncation gets stuck on `LOG_BACKUP` waits
— trades point-in-time recovery for operational stability.

## Auto-schema reconciliation

On startup `app/database/session.py:reconcile_columns()` runs against
the system DB:

1. For every model in `Base.metadata`:
   - If the table doesn't exist, skip (Base.metadata.create_all owns that).
   - If a model column doesn't exist in the DB → `ALTER TABLE ADD COLUMN`.
   - If a model says nullable but the DB says NOT NULL → widen it.

This closes the gap that `create_all(checkfirst=True)` leaves —
`create_all` creates *missing tables* but never adds new columns to
existing ones.

### What this means for you

- Adding a new column to a SQLAlchemy model **just works** on next
  startup — no manual ALTER needed.
- Adding a new table needs `Base.metadata.create_all(...)` to fire,
  which it does on startup.
- Renaming a column is **not** auto-migrated — that needs a manual
  migration script.

## How to add a new table to the data DB

### Option A — via the Create Table UI

For one-off / admin-controlled tables. Doesn't make a Python model.

### Option B — via a SQLAlchemy model (preferred for app-code-managed tables)

1. `app/models/<domain>.py` — declare the model:
   ```python
   from sqlalchemy import Column, Integer, String, DateTime, func
   from app.database.session import DataBase  # or Base for system DB

   class MyTable(DataBase):
       __tablename__ = "ARS_MY_TABLE"
       id = Column(Integer, primary_key=True)
       name = Column(String(200), nullable=False)
       created_at = Column(DateTime, server_default=func.now())
   ```
2. Make sure the module is imported somewhere reachable from
   `main.py` so `Base.metadata` knows about it.
3. Restart backend → `create_all` creates it; `reconcile_columns`
   keeps it in sync afterwards.

## Performance: indexes

Index everything you filter or join on. Common patterns:

```sql
-- Date-based reports
CREATE INDEX IX_Trend_Sales_Daily_RD ON Trend_Sales_Daily(REPORT_DATE);

-- Allocation lookups
CREATE INDEX IX_AlcDetail_Code ON ARS_ALLOCATION_DETAIL(allocation_code);

-- pend_alc filters
CREATE INDEX IX_pend_alc_RDC_ST ON ARS_pend_alc(RDC, ST_CD);

-- Audit drill-downs
CREATE INDEX IX_audit_log_table_batch ON audit_log(table_name, batch_id, created_at);
```

Watch for these anti-patterns:

- `CAST(date_col AS DATE) = '2026-04-29'` → non-sargable, full scan.
  Fix: `date_col >= '2026-04-29' AND date_col < '2026-04-30'`.
- `LIKE '%foo%'` on indexed columns → full scan.
- `OR` across multiple indexed columns → often forces a scan;
  rewrite as `UNION ALL`.

## RCSI — the lock-killer setting

`READ_COMMITTED_SNAPSHOT` should be **on** for both DBs. With it,
readers use row-versioning instead of shared locks; writers don't
block readers and vice versa. Verify:

```sql
SELECT name, is_read_committed_snapshot_on
FROM sys.databases WHERE name IN ('Claude','Rep_Data');
```

Both should show `1`. The startup `enable_rcsi()` tries to flip it on
but Azure SQL DB usually has it pre-enabled.

## Diagnostic SQL you'll need eventually

```sql
-- What's blocking right now?
SELECT session_id, blocking_session_id, wait_type, wait_resource,
       cpu_time, total_elapsed_time
FROM sys.dm_exec_requests
WHERE blocking_session_id <> 0;

-- Long-running open transactions (the 9002 culprits)
SELECT s.session_id, s.host_name, s.program_name,
       at.transaction_begin_time,
       DATEDIFF(SECOND, at.transaction_begin_time, GETDATE()) AS age_sec
FROM sys.dm_tran_active_transactions at
JOIN sys.dm_tran_session_transactions st ON at.transaction_id = st.transaction_id
JOIN sys.dm_exec_sessions s ON st.session_id = s.session_id
WHERE s.is_user_process = 1
ORDER BY at.transaction_begin_time;

-- Top expensive queries
SELECT TOP 10 query_text =
       SUBSTRING(t.text, (qs.statement_start_offset/2)+1,
                 ((CASE qs.statement_end_offset WHEN -1 THEN DATALENGTH(t.text)
                   ELSE qs.statement_end_offset END - qs.statement_start_offset)/2)+1),
       qs.execution_count, qs.total_elapsed_time / qs.execution_count AS avg_us
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) t
ORDER BY avg_us DESC;
```
