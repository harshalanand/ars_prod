# Alembic — DB Schema Migrations for ARS

Alembic manages schema changes to the **System DB (Claude)** — RBAC, RLS, Audit,
TableRegistry, retail master tables, etc. — so that DEV and PROD stay in sync
through git.

> The **Data DB (Rep_data)** is not managed here. Its tables (uploads, MSA
> output, dynamic grids) are created at runtime by raw SQL. Don't put those
> in Alembic; they're owned by the services that create them.

---

## TL;DR — daily workflow

```bash
# in DEV, after you changed a SQLAlchemy model (added column, new table, ...)
cd backend
alembic revision --autogenerate -m "add phone_number to users"
# REVIEW the generated file in backend/alembic/versions/ before committing
alembic upgrade head           # applies to your local DEV DB
git add alembic/versions/ && git commit -m "migration: phone_number on users"
git push

# on PROD, after pulling
cd backend
alembic upgrade head           # applies any new migrations to prod DB
# restart the app
```

That's it. **Never edit the prod DB by hand.** Every schema change goes through
a migration file in git.

## One-time setup (for an already-deployed prod DB)

Prod already has all current tables (created historically by
`Base.metadata.create_all`). We need to tell Alembic "prod is at version
`<initial>` — don't try to re-create those tables."

### Step 1 — install alembic in your environment

```bash
cd backend
pip install -r requirements.txt
```

### Step 2 — back up both DBs

In Azure Portal: `ars-v2retail-sql` → Databases → `Claude` & `Rep_data` →
**Export** → save the `.bacpac` files. Do this **before** any migration work.

### Step 3 — generate the baseline migration (DEV only)

Point your `.env` at a **fresh empty DB** (or use a local SQL Server with no
ARS tables), then:

```bash
cd backend
alembic revision --autogenerate -m "baseline_initial_schema"
```

This produces something like `backend/alembic/versions/abc123_baseline_initial_schema.py`
containing `op.create_table(...)` calls for every model currently in
`app.models`. Review it carefully — it should match the schema you already
have in prod.

Commit it.

### Step 4 — stamp DEV and PROD

On both DBs (in turn — set `.env` appropriately):

```bash
alembic stamp head
```

`stamp` writes the current revision into the `alembic_version` table
**without running the migration**. From now on, Alembic knows "this DB is
already at HEAD." All future `upgrade head` calls run only what comes after.

## Making schema changes (every time after setup)

1. **Edit the SQLAlchemy model** in `backend/app/models/*.py` — add column,
   add table, change type, etc.

2. **Autogenerate the migration**:
   ```bash
   cd backend
   alembic revision --autogenerate -m "short description"
   ```

3. **Review the generated file** at `backend/alembic/versions/<rev>_<slug>.py`.
   Autogenerate is not perfect — it can miss:
   - Server defaults (use `server_default=`, not `default=`, if you need them
     applied at the DB level for existing rows).
   - Renames (it sees them as drop+add — fix manually with `op.alter_column(...,
     new_column_name=...)`).
   - Check constraints on MSSQL (sometimes needs `op.execute(...)`).
   And it may include:
   - Spurious drops of tables it doesn't know about. If you see one, add the
     table name to `IGNORED_TABLES` in `env.py`.

4. **Test on DEV first**:
   ```bash
   alembic upgrade head
   # exercise the feature, check the table looks right
   alembic downgrade -1     # confirm downgrade works
   alembic upgrade head     # back to HEAD
   ```

5. **Commit + push the migration file** alongside the model change. They
   travel together. Never push one without the other.

6. **On PROD**, after `git pull`:
   ```bash
   cd backend
   alembic upgrade head
   # restart the FastAPI app (so SQLAlchemy session reflects the new schema)
   ```

## Useful commands

```bash
alembic current              # show the current revision applied to this DB
alembic history              # list all migrations
alembic heads                # show the unmerged tips (should be 1 in a healthy repo)
alembic show <rev>           # show one migration's contents
alembic upgrade head         # apply all pending
alembic upgrade +1           # apply next migration only
alembic downgrade -1         # roll back one migration
alembic downgrade base       # roll back ALL — dangerous, dev only
alembic upgrade head --sql   # print SQL, don't run it (review-before-apply)
```

## Rules

- **Never edit prod manually.** Drift breaks autogenerate.
- **Always review autogenerated migrations** before committing. Read the diff.
- **Test the migration on a copy of prod data** for risky changes
  (data backfills, NOT NULL adds, type changes on big tables).
- **Data migrations are two steps:** add column NULL → backfill → switch code
  → make NOT NULL. Don't try to do it in one migration on a populated table.
- **`alembic stamp head` on an empty new DB is wrong** — it skips the create.
  Only stamp DBs that already have the schema.
- **One head**: if `alembic heads` shows two, you have a branched history.
  Merge with `alembic merge -m "merge" <rev1> <rev2>`.

## Targeting a different DB ad-hoc

For one-off operations against a DB that isn't your `.env`:

```bash
alembic -x url="mssql+pyodbc://user:pass@host/Claude?driver=ODBC+Driver+18+for+SQL+Server" upgrade head
```

The `-x url=...` overrides `settings.DATABASE_URL` for this one command.

## Coexistence with `Base.metadata.create_all`

`backend/main.py` still calls `Base.metadata.create_all(checkfirst=True)` at
startup. That stays for now — it's idempotent and only creates **missing**
tables, so it won't fight Alembic. The split:

- **`create_all`** — safety net for brand-new tables that Alembic somehow
  missed on this environment.
- **Alembic** — the source of truth for every schema change going forward.

Once you've run through one full DEV→PROD cycle with Alembic and trust it,
you can remove the `create_all` call from `main.py`. Not urgent.
