## TAB 1 — Manual Entry (`POST /pend-alc/manual-upload`)

### 1a. How it works today — end to end

**Endpoint:** `pend_alc.py:2245-2343`. Body: `ManualUploadRequest` (`pend_alc.py:2233-2242`).

Flow:

1. **Pydantic decode** (`pend_alc.py:2267`) — `body.rows` → list of dicts. No domain validation; only structural typing (`rdc: str`, `alloc_qty: float`, all others optional incl. `st_cd`, `maj_cat`, `gen_art_number`, `clr`, `remarks`).
2. **`write_manual_pend_alc(conn, rows, session_id=body.session_id)`** (`pend_alc_service.py:2335`):
   - Filters `alloc_qty > 0` — silently drops zero or negative qty rows (`pend_alc_service.py:2361`).
   - If `session_id` is None, builds one `f"MANUAL-{YYYYMMDD}-{6hex}"` (`pend_alc_service.py:2367-2370`).
   - Captures `prev_max_id = ISNULL(MAX(ID),0) WHERE SESSION_ID=:sid` BEFORE the insert so the post-insert ID readback only picks up THIS call's rows (`pend_alc_service.py:2376-2378`). Important for chunked uploads.
   - Bulk insert via `pyodbc.fast_executemany`, internal `CHUNK=10000` (`pend_alc_service.py:2407-2427`). Column order:
     ```
     (SESSION_ID, RDC, ST_CD, ARTICLE_NUMBER, MAJ_CAT, GEN_ART_NUMBER, CLR,
      ALLOC_QTY, ALLOC_MODE, SOURCE, REMARKS)
     VALUES (?, ?, ?, ?, ?, ?, ?, ?, 'MANUAL', 'MANUAL', ?)
     ```
     Note `ALLOC_MODE='MANUAL'` and `SOURCE='MANUAL'` are hard-coded. `BDC_QTY`, `DO_QTY`, `IS_CLOSED`, `APPROVED_AT` rely on DDL defaults (0, 0, 0, GETDATE()).
   - `conn.commit()` then a readback to compute `inserted_ids = [ID WHERE SESSION_ID=:sid AND ID > :prev]`.
3. **Deferred delta** (`pend_alc.py:2285-2310`) — ONLY runs when `body.is_last_chunk=True`. Re-reads ALL rows for the session from `ARS_PEND_ALC` (not just this chunk) and feeds them to `apply_pend_alc_delta(conn, all_rows, sign=+1)`. This is the only path that propagates the +1 PEND_QTY to `ARS_MSA_TOTAL/VAR_ART/GEN_ART` and to every `ARS_GRID_MJ*` rollup.
4. **`log_operation_upsert`** (`pend_alc.py:2315-2329`, `pend_alc_service.py:250`):
   - Chunk 1: INSERT into `ARS_PEND_ALC_OPERATIONS` (`OP_TYPE='MANUAL'`, `OP_KEY=session_id`, full payload).
   - Chunk N>1: UPDATE the existing row by `(OP_TYPE, OP_KEY)`, adding `rows_affected` and `qty_total`. **No `merge_payload_lists` passed**, so the PAYLOAD JSON stays at chunk-1's value — `inserted_ids` and `article_rdc_pairs` from chunks 2..N are **not stored**.
5. Response: `{success, inserted_rows, session_id, msa_adjusted}`.

**Delta details** (`pend_alc_service.py:4803-5085`):
- Creates `#delta_<hex>` with collation pinned to `DATABASE_DEFAULT` (avoids SQL 468 collation conflict — comment at 4843-4848 is real and necessary).
- Aggregates by `(rdc, art)` for `ARS_MSA_TOTAL`. PEND_QTY incremented; FNL_Q recomputed as `max(STK − new_PEND − HOLD, 0)`.
- `ARS_MSA_VAR_ART`/`GEN_ART` copy/sum from MSA_TOTAL — UPDATE-only.
- `_discover_grid_rollup_tables(conn)` → every active `ARS_GRID_MJ*` rollup; the +1 path also `_build_rollup_insert_sql` to add placeholder rows for new (WERKS, hier) tuples.

**Indexes hit:**
- INSERT: clustered PK `PK_ARS_PEND_ALC (ID)` plus auto-maintenance of `IX_ARS_PEND_ALC_session (SESSION_ID)` and the others.
- Readback `WHERE SESSION_ID=:sid AND ID > :prev` uses `IX_ARS_PEND_ALC_session`.

### 1b. Inputs & validation

| Field | Required | Defaulted | Validation |
|---|---|---|---|
| `rdc` | yes (Pydantic `str`) | — | Pydantic accepts empty string. Service does NOT reject blank. Insert succeeds with `RDC=''`. |
| `article_number` | yes (Pydantic `str`) | — | Same — accepts `''`. Insert succeeds. |
| `alloc_qty` | yes (Pydantic `float`) | — | `> 0` filter at service (`pend_alc_service.py:2361`). Zero / negative silently dropped, no error returned, `inserted` count drops. |
| `st_cd` | optional | NULL | Normalised by `(str(r.get("st_cd")).strip() if r.get("st_cd") else None) or None` → empty/whitespace becomes NULL. |
| `maj_cat`, `gen_art_number`, `clr` | optional | NULL | Stored as-is. **Never cross-checked against `vw_master_product`**. |
| `remarks` | optional | NULL | Stored as-is, no length truncation (DDL is 500 chars — pyodbc will raise a truncation error if longer). |
| `session_id` | optional | server-generated | No format check; an attacker can pass any string ≤50 chars. |
| `is_first_chunk` / `is_last_chunk` | optional | True / True | Pure trust. No server-side state machine; chunk 5 sent before chunk 1 is fine. |

**Inputs silently accepted that should be rejected:**
- `rdc=''` or `article_number=''` — passes Pydantic, passes service, lands in DB with empty key. MSA delta UPDATE on `T.RDC = '' AND T.[ARTICLE_NUMBER] = ''` will match no rows (no MSA universe entry for blanks), so PEND_QTY never propagates. Row sits forever in `ARS_PEND_ALC` as garbage.
- `alloc_qty <= 0` — silently dropped from staging without reporting in the response. The user sees `inserted_rows=0` and may think the upload failed; service log shows `rows inserted: 0` only at INFO level.
- `(rdc, article_number)` not in `vw_master_product` / `ARS_MSA_TOTAL` (orphan) — accepted. The MSA delta UPDATE matches zero rows. The row is invisible to listing/allocation but counts in Pend Alc dashboards. `bootstrap_msa_pend_sync` won't fix it either because the universe-anchored Step 6 needs the article in `vw_master_product` (KB note: msa.md universe rule).
- `(rdc, gen_art_number)` not in MSA but article exists at a different RDC — accepted. Same outcome: orphan PEND.
- `maj_cat` / `gen_art_number` / `clr` not matching what `vw_master_product` says for this article — accepted and stored. Subsequent grid rollup uses the user-supplied values for grouping, producing miscategorised aggregates.
- Future-dated entries: there is no `EFFECTIVE_DATE` column. `APPROVED_AT` defaults to `GETDATE()`. Can't be backdated.
- Duplicate keys within the same payload — accepted; multiple rows for the same `(RDC, ST_CD, ARTICLE_NUMBER, ALLOC_MODE)` are inserted. The DB has **no unique constraint** on this grain (verified live: only PK on `ID`).

### 1c. Lifecycle effects

**Tables touched:**

| Table | Op | Columns | When |
|---|---|---|---|
| `ARS_PEND_ALC` | INSERT (per chunk) | `SESSION_ID, RDC, ST_CD, ARTICLE_NUMBER, MAJ_CAT, GEN_ART_NUMBER, CLR, ALLOC_QTY, ALLOC_MODE='MANUAL', SOURCE='MANUAL', REMARKS` | every chunk |
| `ARS_MSA_TOTAL` | UPDATE | `PEND_QTY +=`, `FNL_Q = max(STK−PEND−HOLD,0)` | last chunk only |
| `ARS_MSA_VAR_ART` | UPDATE | `PEND_QTY, FNL_Q` (copy from TOTAL) | last chunk only |
| `ARS_MSA_GEN_ART` | UPDATE | `PEND_QTY, FNL_Q` (SUM from TOTAL) | last chunk only |
| `ARS_GRID_MJ_*` (every active rollup) | UPDATE + INSERT (placeholder rows) | `PEND_ALC +=`, hier-resolved attribute columns on missing rows | last chunk only |
| `ARS_PEND_ALC_OPERATIONS` | INSERT (chunk 1), UPDATE rows_affected/qty_total (chunks 2..N) | `OP_TYPE='MANUAL', OP_KEY=session_id, PAYLOAD={session_id, inserted_ids, article_rdc_pairs}` | every chunk |

**Computed vs frozen vs mutable:**
- Frozen at insert: `SESSION_ID, RDC, ST_CD, ARTICLE_NUMBER, MAJ_CAT, GEN_ART_NUMBER, CLR, ALLOC_QTY, ALLOC_MODE='MANUAL', SOURCE='MANUAL', APPROVED_AT=GETDATE()`.
- Computed: `PEND_QTY = ALLOC_QTY − DO_QTY` (PERSISTED computed column, DDL line 84).
- Mutable later: `BDC_QTY, LAST_BDC_AT, DO_QTY, DO_NUMBER, DO_UPLOADED_AT, LAST_DO_AT, IS_CLOSED, REMARKS` — by BDC stamp, DO upload, adhoc close, revert.

**Revert payload sufficiency** (`pend_alc_service.py:1484-1556`, `_revert_manual`):
- Captures `session_id` AND `inserted_ids` (chunk 1 only).
- Revert path prefers `WHERE SESSION_ID = :sid` (covers chunks 2..N even though their IDs were never persisted), falls back to `WHERE ID IN inserted_ids` for legacy entries.
- Reads all matching rows **before** DELETE so it can call `apply_pend_alc_delta(rows, sign=-1)` to back out MSA/grid effects.
- **Sufficient** for normal cases.

**Side effects NOT captured for revert:**
- The `ALLOC_MODE='MANUAL'` and `SOURCE='MANUAL'` are gone after delete — not relevant for revert (rows are erased), but the operations log doesn't keep a copy of the data either. If the original row had a REMARKS appended by adhoc-close BEFORE revert is run (unlikely but possible if order is `manual → adhoc-close → revert manual`), `_check_adhoc_close_revert` and `_check_manual_revert` don't cross-check. The MANUAL revert blocker (`_check_manual_revert:1024`) only flags `BDC_QTY > 0 OR DO_QTY > 0`, not adhoc-closed state.

### 1d. Edge cases and bugs

#### M-1. [HIGH] Duplicate `(RDC, ST_CD, ARTICLE_NUMBER, ALLOC_MODE)` within a session double-count grid rollups
- **file:line:** `pend_alc_service.py:2335-2451` (write), `pend_alc_service.py:5043-5051` (rollup delta).
- **Repro:** payload `rows=[{rdc:'R09', article_number:'A1', alloc_qty:10}, {rdc:'R09', article_number:'A1', alloc_qty:5}]`, no `st_cd`.
- **Starting state:** `ARS_MSA_TOTAL[R09,A1].PEND_QTY=0`, `ARS_GRID_MJ_VAR_ART[R09,A1].PEND_ALC=0`. No prior PEND row.
- **Action:** call manual upload. Both rows are inserted (no unique constraint on the grain — verified live: `rep_data.sys.indexes` for `ARS_PEND_ALC` has no unique index covering session+rdc+st_cd+article+mode).
- **Resulting:** `ARS_MSA_TOTAL.PEND_QTY=15` (aggregated by `(rdc,art)` correctly), `ARS_GRID_MJ_VAR_ART.PEND_ALC=15` (rollup aggregates correctly). Two PEND_ALC rows with `ALLOC_QTY=10` and `ALLOC_QTY=5`.
- **Impact:** Actually OK for end-state numbers because every aggregator collapses by `(rdc, art)`. BUT downstream effects diverge: `bdc-generate` will create TWO history rows (one per pend row) at FIFO order, doubling the BDC entry count. SAP DO matching can FIFO-credit them out of input order. Confusing reco UI.
- **Severity:** High — not a numeric leak, but a "one upload, many lifecycle artefacts" surprise.

#### M-2. [HIGH] Missing `is_last_chunk=True` orphans the MSA/grid delta
- **file:line:** `pend_alc.py:2285-2310`.
- **Repro:** Frontend sends 5 chunks with `is_last_chunk=False` on every chunk (bug, mid-upload kill, network drop on the final chunk, …).
- **Starting state:** Any.
- **Resulting:** All 5 chunks' rows are in `ARS_PEND_ALC`. The delta runs 0 times. `ARS_MSA_TOTAL.PEND_QTY` and `ARS_GRID_MJ*.PEND_ALC` stay at their pre-upload values. The next listing/allocation run sees inflated FNL_Q → over-allocates → real WH ships short.
- **Detection:** the `bootstrap_msa_pend_sync` only runs on (a) MSA Generate, (b) post-revert, (c) post-grid-build. Until one of those fires, the inflated FNL_Q stays. The dashboard `/pend-alc/pend-vs-msa-gap` will surface it, but no auto-heal.
- **Severity:** High — silent stock leak proportional to upload size.

#### M-3. [HIGH] Sending `is_last_chunk=True` on multiple chunks runs the delta multiple times
- **file:line:** `pend_alc.py:2285-2310`.
- **Repro:** Frontend bug or retry: chunk 2 of 3 is re-sent because of a transient 502, marked `is_last_chunk=True` on the retry to "be safe".
- **Starting state:** Session `S1` has rows from chunk 1 (10 rows, sum=100) and chunk 2 (10 rows, sum=200) successfully inserted.
- **Action:** Chunk 2 retry arrives with `is_last_chunk=True` (this isn't actually the last chunk).
- **Resulting:** The endpoint runs `apply_pend_alc_delta(all_rows, sign=+1)` reading the union of chunks 1+2 — adds +300 to PEND_QTY. Chunk 3 arrives later with `is_last_chunk=True` and runs the delta AGAIN with ALL chunks 1+2+3 — adds another +600 (the full union, not just chunk 3). Net effect: `PEND_QTY` over-stated by +300.
- **Impact:** Stock under-allocation (FNL_Q deflated). Reverting via `_revert_manual` applies `−1 × (alloc_qty − do_qty)` for every PEND row exactly once, so revert restores the correct value — but the time window between the buggy upload and revert sees wrong FNL_Q.
- **Severity:** High — there is no idempotency token on the delta.

#### M-4. [MEDIUM] `apply_pend_alc_delta` errors mid-flight leave PEND_ALC out of sync with MSA
- **file:line:** `pend_alc.py:2266-2343`.
- **Repro:** A collation conflict slips past the `DATABASE_DEFAULT` pin (e.g. `MAJ_CAT` longer than 200 chars truncates inside the temp table), or `_discover_grid_hierarchy` raises on a malformed grid table.
- **Resulting:** The endpoint's outer `try` (`pend_alc.py:2266`) catches and re-raises as `HTTPException(500)`. The PEND_ALC rows are already committed (line 2431 commits inside `write_manual_pend_alc`). MSA/grid never updated. User retries — duplicates land. No transactional boundary around the full chunk.
- **Impact:** Manual upload requires a redo, but redo without de-dup creates M-1.
- **Severity:** Medium.

#### M-5. [HIGH] Orphan `(RDC, ARTICLE)` not in MSA universe sits forever
- **file:line:** `pend_alc.py:2245`, `pend_alc_service.py:4932-4953`.
- **Repro:** User uploads `{rdc:'R09', article_number:'9999999', alloc_qty:5}` where article 9999999 is not in `vw_master_product` and not in `ARS_MSA_TOTAL`.
- **Resulting:** Row inserted. `apply_pend_alc_delta` MSA_TOTAL UPDATE matches 0 rows (the LEFT JOIN target doesn't exist). The +1 grid rollup INSERT (lines 5043-5046) MAY create a placeholder grid row, but with NULL hier attributes resolved from `vw_master_product` (which has no row) — so the grid row's `FAB`, `MICRO_MVGR`, etc., are NULL. That breaks sec-cap propagation (invariant 4).
- **Impact:** Sec-cap grids silently corrupted for orphan articles. Listing/allocation runs against these grids skip them or treat them as a uniform null-grain bucket.
- **Severity:** High — direct invariant violation, no validation gate.

#### M-6. [MEDIUM] Manual entry against an article that has an open AUTO PEND for the same key
- **file:line:** `pend_alc_service.py:2335`.
- **Repro:** Auto-allocation Session `A1` already created `(SESSION_ID='A1', RDC='R09', ST_CD='S01', ART='A1', ALLOC_MODE='RL', ALLOC_QTY=20, IS_CLOSED=0)`. User adds manual entry `(RDC='R09', ST_CD='S01', ART='A1', alloc_qty=5)`.
- **Resulting:** Second row inserted with `ALLOC_MODE='MANUAL', SOURCE='MANUAL'`. Total PEND_QTY for the key becomes 25. No warning. SAP DO arriving for this key will FIFO-consume the older AUTO row first (`APPROVED_AT, ID` partition — see DO #3 below) before touching MANUAL.
- **Impact:** Operationally fine — both pend reductions matter — but the user is unaware the original auto-row exists. If they then adhoc-close the article (M-D5 below) the manual row AND the auto row both get IS_CLOSED=1 plus REMARKS prefix.
- **Severity:** Medium — surprise factor.

#### M-7. [MEDIUM] Manual entry coexisting with parked listing for the same key
- **file:line:** No cross-check.
- **Repro:** Listing is parked (rows in `ARS_NL_TBL_HOLD_TRACKING` or `ARS_LISTING_*PARKED`) for `(R09, S01, A1)`. User manually uploads PEND for the same key.
- **Resulting:** Manual row inserted, MSA PEND_QTY goes up. Subsequent approve_parked of the parked session will ADD ITS OWN PEND on top via `write_pend_alc` → MSA double-counts until both are closed.
- **Severity:** Medium — would require a parked-vs-pend cross-check the system doesn't currently have.

#### M-8. [LOW] `is_first_chunk=False` on the very first chunk creates duplicate ops_log rows
- **file:line:** `pend_alc_service.py:328-353` (`log_operation_upsert` fallback).
- **Repro:** Frontend bug: first chunk sent with `is_first_chunk=False`.
- **Resulting:** The `UPDATE WHERE OP_TYPE='MANUAL' AND OP_KEY=session_id` matches 0 rows. The code falls through to `log_operation` and INSERTs a fresh row. Chunk 2 (sent correctly with `is_first_chunk=False`) UPDATEs that row. End-state: one row, but `rows_affected` from chunk 1 was recorded as "+initial INSERT", chunk 2 added "+rows", **chunk 1's rows_affected is counted once and chunk 2 is added on top**. Net `rows_affected` = total. OK.
- The risk: if two distinct uploads happen to share the same session_id by user error (next M-9), the second upload's chunk 1 falls through to INSERT a second op row. Now there are two operations log rows for the same session_id, both active. Revert from the UI picks one; the other is orphaned.
- **Severity:** Low — possible but requires a session_id collision.

#### M-9. [HIGH] Reusing a `session_id` across two different upload events merges them as one revertable unit
- **file:line:** `pend_alc.py:2274` (no validation), `pend_alc_service.py:2376`.
- **Repro:** User on Tab A uploads manual entries with `session_id='S1'` (10 rows). 30 minutes later, on Tab B (or different machine), uploads a different file with the same `session_id='S1'` (50 rows).
- **Starting state:** After upload 1, `ARS_PEND_ALC` has 10 rows with `SESSION_ID='S1'`; ops_log has one MANUAL row with `OP_KEY='S1'`.
- **Action:** Upload 2 runs `write_manual_pend_alc(conn, rows, session_id='S1')`. The `prev_max_id` snapshot captures upload-1's max ID; only upload-2's IDs are returned. Then `log_operation_upsert(is_first=True)` is called — but since `is_first=True`, it goes straight to `log_operation` which INSERTs a NEW ops_log row.
- **Resulting:** 60 PEND rows under SESSION_ID='S1'. TWO ops_log rows, both with OP_KEY='S1'. **The deferred delta** runs on `WHERE SESSION_ID=:sid` which reads all 60 rows — so MSA gets +60 row qty when only +50 was intended for upload-2 (upload-1's MSA was already applied 30 min earlier).
- **Impact:** MSA over-deducts by upload-1's qty. Reverting either ops_log row deletes ALL 60 rows via `WHERE SESSION_ID=:sid` (`_revert_manual:1507`) and the other ops_log row's revert becomes a no-op delete + still-stale ops_log entry.
- **Severity:** High — concrete double-counting + orphan ops_log entries on session_id reuse. There's no de-dup of `session_id` at INSERT time.

#### M-10. [LOW] Empty vs NULL `ST_CD`
- **file:line:** `pend_alc_service.py:2385`.
- **Behaviour:** `(str(r.get("st_cd")).strip() if r.get("st_cd") else None) or None` collapses both `""` and `None` to `NULL`. Consistent.
- The grid hierarchy normalises with `ISNULL(P.ST_CD,'')` everywhere, so this is fine.

#### M-11. [MEDIUM] `ALLOC_MODE` field on `ManualRow` doesn't exist — always hard-coded to `'MANUAL'`
- **file:line:** `pend_alc_service.py:2411`.
- The `ManualRow` Pydantic model has no `alloc_mode` field, and even if it did, the INSERT VALUES literal `'MANUAL'` would ignore it. Users cannot tag a manual row as RL/TBC/TBL. Means manual rows are excluded from FIFO partition rotation: in `apply_do_deductions` the partition is `(RDC, ST_CD, ART)` only — ALLOC_MODE doesn't gate it. But in the BDC view, manual rows show up under their own mode column, so reports filtered by mode='RL' will hide them.

## TAB 2 — Daily DO Entry (`POST /pend-alc/do-update` and `/do-update-async`)

### 2a. How it works today — end to end

**Sync endpoint:** `pend_alc.py:560-656`. Body: `DoUpdateRequest` (`pend_alc.py:548`).
**Async endpoint:** `pend_alc.py:1351-1378`, worker `_do_run_job` (`pend_alc.py:1239-1349`).

Flow (sync path):

1. **Pydantic decode** (`pend_alc.py:583-596`) — each row: `rdc, article_number, do_qty, allocation_number?, st_cd?, do_number?`.
2. Generate `session_id` if not supplied: `uuid.uuid4().hex[:12]` (`pend_alc.py:589`).
3. **`apply_do_deductions(conn, rows)`** (`pend_alc_service.py:3192-3515`):
   - Filters `do_qty > 0` for the apply path (`pend_alc_service.py:3240`). **`do_qty == 0` is dropped here**, but the `update_bdc_history_with_do` call below treats `qty=0` as a cancel signal.
   - Creates `#do_in_<hex>` and bulk-loads input via `fast_executemany` with `(seq, rdc, st_cd, art, alloc_no, qty, do_num)`.
   - **Allocation-number resolve pass** (`pend_alc_service.py:3316-3331`): For input rows carrying `alloc_no`, looks up `ARS_BDC_HISTORY.ST_CD` for that `(alloc_no, rdc, art)`; BDC's ST_CD overwrites user-supplied ST_CD. If no BDC matches, user's ST_CD is kept.
   - Creates `#do_out_<hex>` for OUTPUT capture (per-row deltas).
   - **Two-pass FIFO** (`pend_alc_service.py:3346-3418`):
     - Scoped pass (`scope='scoped'`): `agg.st_cd <> ''` and join `agg.st_cd = ISNULL(o.ST_CD,'')`.
     - Global pass (`scope='global'`): `agg.st_cd = ''` and join `1=1`.
     - Each pass aggregates `tmp_in` by `(rdc, st_cd, art)` (also `STRING_AGG(do_num)` in `seq` order for the `DO_NUMBER` concat).
     - Open PEND rows ranked by `PARTITION BY P.RDC, ISNULL(P.ST_CD,''), P.ARTICLE_NUMBER ORDER BY P.APPROVED_AT, P.ID`. Running-sum CTE for cumulative `need`.
     - `alloc_plan` computes per-row `apply_qty = clamp(agg.qty − cum_before, [0, need])`.
     - One UPDATE per pass:
       ```
       SET DO_QTY = DO_QTY + apply_qty,
           IS_CLOSED = CASE WHEN DO_QTY + apply_qty >= ALLOC_QTY THEN 1 ELSE 0 END,
           LAST_DO_AT = GETDATE(),
           DO_UPLOADED_AT = GETDATE(),
           DO_NUMBER = concat
       ```
     - OUTPUT captures `INSERTED.ID, qty_added, was_just_closed, prev_last_do_at` into `#do_out`.
   - **Auto-close BDC history** when a PEND row just-closed (`pend_alc_service.py:3447-3487`): for every `(RDC, ST_CD, ARTICLE)` of a just-closed PEND row, flip every `STATUS='OPEN'` BDC_HISTORY row to `'CONFIRMED'`. Captured to `auto_history_closes` list.
   - Returns `{touched, pend_updates, auto_history_closes}`.
4. **`update_bdc_history_with_do(conn, rows)`** (`pend_alc_service.py:2566-2875`):
   - Buckets input by routing: `by_alloc` (alloc_no provided, qty>0), `by_store` (st_cd provided, no alloc_no, qty>0), `by_global` (only rdc+art, qty>0), plus three cancel buckets for `qty=0`.
   - Cancellation passes run FIRST (so a user can both cancel and reapply in one upload). Each cancel pass `UPDATE H SET STATUS='CANCELLED' WHERE STATUS='OPEN'` joined to the agg key. **Pre-image** captured into `tmp_out`.
   - Then three apply passes (alloc, store, global) — each a windowed FIFO UPDATE on `ARS_BDC_HISTORY`. `STATUS` set to `'CONFIRMED'` if fully covered else `'CLOSED_PARTIAL'`. `LAST_DO_AT = GETDATE()`.
   - Returns `{touched, history_updates}` with full pre/post images.
5. **`log_operation_upsert`** (`pend_alc.py:609-635`):
   - `merge_payload_lists=["pend_updates","history_updates","auto_history_closes"]` — chunks 2..N read-modify-write into the chunk-1 payload so revert covers everything.
   - `OP_TYPE='DO'`, `OP_KEY=session_id`.
6. Response: `{success, updated_rows, bdc_history_updated, bdc_cancelled, session_id, do_batch_id}`.

**Async worker** (`pend_alc.py:1239-1349`):
- Receives the FULL row list in one POST. Slices internally at `_DO_SLICE=25_000` (`pend_alc.py:1236`).
- For each slice: `apply_do_deductions(conn, sl)` + `update_bdc_history_with_do(conn, sl)` — runs INSIDE one `with _engine().connect()` block, NOT one transaction (each function commits internally).
- Aggregates `pend_updates`, `history_updates`, `auto_history_closes` in-process across slices.
- ONE final `log_operation_upsert(is_first=True)` with the aggregate.
- Crucially: if slice 3/5 errors, `_job_update(status='failed')` is set, but slices 1–2 are already committed to the DB. **No rollback of partial DB writes.** The ops_log row is never written. So the DO is applied but unauditable / unrevertable.

**Indexes hit:**
- PEND_ALC FIFO walk uses `IX_ARS_PEND_ALC_bdc_lookup (IS_CLOSED, RDC, ST_CD, ARTICLE_NUMBER) INCLUDE (PEND_QTY, BDC_QTY, MAJ_CAT, LAST_BDC_AT)` — good seek.
- BDC history FIFO uses `IX_ARS_BDC_HISTORY_lookup (RDC, ST_CD, ARTICLE_NUMBER)` + `IX_ARS_BDC_HISTORY_alloc (ALLOCATION_NUMBER)`.

### 2b. Inputs & validation

| Field | Required | Defaulted | Validation |
|---|---|---|---|
| `rdc` | yes | — | Pydantic accepts empty; service strips, requires truthy via `if not rdc or not art or qty < 0: continue` (`pend_alc_service.py:2632`). Blank silently skipped. |
| `article_number` | yes | — | Same. |
| `do_qty` | yes | — | `>= 0` filter in apply path (`< 0` skipped in history path). Negative silently dropped. |
| `allocation_number` | optional | — | No format check. |
| `st_cd` | optional | — | Empty string treated as "no scope". |
| `do_number` | optional | — | Concatenated into `DO_NUMBER` via STRING_AGG. No length check (DDL is 100 chars; very long input across many chunks for same PEND row will eventually overflow). |
| `session_id`, `is_first_chunk`, `is_last_chunk` | optional | server-gen / True / True | Same as manual. |

**Silently accepted that should be rejected:**
- Negative `do_qty` — `< 0` filtered out silently.
- DO for `(RDC, ARTICLE)` where there is no open PEND row AND no open BDC — both `apply_do_deductions` (no open ranked rows) and `update_bdc_history_with_do` (no agg match) become no-ops. Row count reported as 0. **No warning to the user.** Looks like a successful upload, but nothing happened.
- DO for an article that's adhoc-closed: `apply_do_deductions` filters `WHERE P.IS_CLOSED=0` so the closed PEND is skipped. `update_bdc_history_with_do` filters `WHERE H.STATUS='OPEN'` — the adhoc-cancel already flipped to CANCELLED, so this also skips. End result: silent no-op.
- `allocation_number` that doesn't exist in BDC_HISTORY — the OUTER APPLY on `pend_alc_service.py:3320` returns NULL, the row falls through to FIFO. **No error**. If the user intended to settle a specific BDC and typed it wrong, they get cross-attribution to a different open PEND.
- Duplicate `(RDC, ST_CD, ARTICLE)` rows within a single payload — explicitly handled: `agg` aggregates with `SUM(qty)`. Fine.
- DO for a future DO date — there is no DO date input; `LAST_DO_AT=GETDATE()` always.

| Table | Op | Columns | When |
|---|---|---|---|
| `ARS_PEND_ALC` | UPDATE | `DO_QTY +=, IS_CLOSED, LAST_DO_AT, DO_UPLOADED_AT, DO_NUMBER` | every chunk |

| `ARS_PEND_ALC_OPERATIONS` | INSERT (chunk 1) or UPDATE+merge_payload (chunks 2..N sync) | `OP_TYPE='DO'` with `{pend_updates, history_updates, auto_history_closes}` | every chunk (sync) / once (async) |
| **`ARS_MSA_TOTAL/VAR_ART/GEN_ART`** | **NOT TOUCHED** | — | by design — STK_QTY is daily snapshot |
| `ARS_NL_TBL_HOLD_TRACKING` | NOT touched | — | by design — hold release decoupled |

**Computed vs frozen vs mutable:**
- Mutated only: `DO_QTY, IS_CLOSED, LAST_DO_AT, DO_UPLOADED_AT, DO_NUMBER` on PEND_ALC; `DO_RECEIVED, STATUS, LAST_DO_AT` on BDC_HISTORY.
- `PEND_QTY` recomputes via the PERSISTED computed column.

**Revert payload sufficiency** (`pend_alc_service.py:1305-1481`, `_revert_do`):
- `pend_updates` carries `{pend_alc_id, qty_added, was_just_closed, prev_last_do_at}`.
- `history_updates` carries `{history_id, qty_added, old_do_received, new_do_received, old_status, new_status, old_last_do_at}`.
- `auto_history_closes` carries `{history_id, old_status, prev_last_do_at}`.
- Revert subtracts `qty_added` from DO_QTY, restores `IS_CLOSED=0` if `was_just_closed=1`, restores `LAST_DO_AT`, restores BDC_HISTORY status/last_do_at.
- **NOT captured for revert:**
  - `DO_UPLOADED_AT` — not in payload. Revert restores `LAST_DO_AT` but `DO_UPLOADED_AT` stays at the post-apply timestamp.
  - `DO_NUMBER` — the STRING_AGG appended this DO's `do_num` to `P.DO_NUMBER`. Revert never strips it. After revert, `DO_NUMBER` still shows the no-longer-applicable DO number. Visible in reco / dashboard.

### 2d. Edge cases and bugs

#### D-1. [CRITICAL] FIFO partition omits `ALLOC_MODE` → cross-attribution between RL/TBC/TBL
- **file:line:** `pend_alc_service.py:3371, 3376` and confirmed on `2724`.
- **Repro:** PEND_ALC has two open rows for `(R09, S01, A1)`: ID=100 (`ALLOC_MODE='RL'`, ALLOC_QTY=10, APPROVED_AT=Jun 1) and ID=200 (`ALLOC_MODE='TBC'`, ALLOC_QTY=15, APPROVED_AT=Jun 5).
- **Action:** SAP DO arrives for `(R09, S01, A1, do_qty=5)` from a TBC BDC.
- **Resulting:** FIFO ranks by `APPROVED_AT, ID` → ID=100 (RL) consumes first. After UPDATE: ID=100 has DO_QTY=5 (RL closed by 50%); ID=200 (TBC) unchanged. The TBC line that actually shipped is unsettled in the data model.
- **Impact:**
  - RL/TBC/TBL reporting becomes wrong — RL allocations show as fulfilled when they aren't, TBC allocations show as pending when they're shipped.
  - The next BDC cycle picks the wrong source: RL row's residual is 5, may not retrigger BDC; TBC row still shows PEND_QTY=15 and will get re-BDCed → double-BDC.
- **Severity:** Critical — direct invariant violation (OPT_TYPE uniqueness, invariant 1). Confirms the prior audit's #8 with SQL trace.

#### D-2. [HIGH] DO qty exceeding open PEND silently caps and reports zero excess
- **file:line:** `pend_alc_service.py:3389-3391`.
- **Repro:** Open PEND for `(R09, S01, A1)` has total `need=10` (one row, ALLOC_QTY=10, DO_QTY=0). SAP DO arrives with `do_qty=25`.
- **Resulting:** `agg.qty=25`, `cum_after=10`. `apply_qty = min(25, 10) = 10`. PEND row closed. **No record of the 15-unit overflow** — neither in PEND_ALC nor in BDC_HISTORY nor in ops_log. The user sees `updated_rows=1, bdc_history_updated=N`. No warning that 15 units of DO had nowhere to land.
- **Impact:** Reality is that 25 units shipped; the system thinks 10. MSA never reconciles the 15 because DO doesn't touch MSA at all. Next BDC will see PEND_QTY=0 for this key (correct), so no double-BDC. But warehouse stock-out from MSA's STK side is not visible until next full MSA Generate refreshes STK_QTY. Net: warehouse-vs-MSA drift between DO and next MSA run.
- **Severity:** High — silent data loss with no audit trail.

#### D-3. [HIGH] DO arriving for a key with no open BDC (BDC never stamped)
- **file:line:** `pend_alc_service.py:2566` (history pass), `pend_alc_service.py:3192` (pend pass).
- **Repro:** Manual upload created PEND_ALC row, BDC was never generated. SAP DO arrives.
- **Resulting:** `apply_do_deductions` walks PEND_ALC and consumes it correctly (BDC is irrelevant to PEND path). `update_bdc_history_with_do` finds no matching `STATUS='OPEN'` row → all three passes are no-op. The PEND row is closed; the BDC_HISTORY has no record of this DO.
- **Impact:** The `bdc_history` audit trail has a gap. Reco UI shows the PEND row as DO'd but with no allocation_number / bdc_qty match. The downstream "BDC IN FLIGHT" tile is unaffected (correct) but "DO vs BDC" reconciliation is wrong.
- **Severity:** High — silent acceptance of an out-of-order DO. The system should probably reject or quarantine.

#### D-4. [HIGH] Multi-chunk sync DO upload crashes mid-chunk → ops_log misses later updates
- **file:line:** `pend_alc.py:560`, `pend_alc_service.py:283-326`.
- **Repro:** Chunks 1 and 2 succeed. Chunk 3 sets `is_first_chunk=False, is_last_chunk=True`. After `apply_do_deductions` commits but before `update_bdc_history_with_do` finishes, the connection drops.
- **Resulting:**
  - PEND_ALC has chunk 3's DO_QTY applied (committed inside `apply_do_deductions`).
  - BDC_HISTORY partially updated (cancel passes may have run; apply passes may not).
  - The outer endpoint's `try/except` catches (`pend_alc.py:655`) and raises HTTPException(500). **`log_operation_upsert` for chunk 3 never executes.**
  - The ops_log row for `session_id=X` still has chunks 1+2's `pend_updates` / `history_updates`. Chunk 3's deltas are NOT in the payload.
  - User reverts: `_revert_do` reads only chunks 1+2's updates. Chunk 3's DB changes stay applied but no longer match `LAST_DO_AT` audit. PEND row may end up with `DO_QTY > ALLOC_QTY` (impossible value).
- **Severity:** High — revert correctness broken by mid-chunk failure.

#### D-5. [CRITICAL] Async worker partial-failure leaves PEND_ALC modified with NO ops_log entry
- **file:line:** `pend_alc.py:1270-1317`.
- **Repro:** Slice 3 of 5 raises a deadlock or timeout.
- **Resulting:** Slices 1–2 are fully committed (PEND_ALC + BDC_HISTORY updated). Exception bubbles out; `_job_update(status='failed')` runs. `log_operation_upsert` at line 1295 is INSIDE the try block, AFTER the loop — so it never runs. **No ops_log row at all.**
- **Impact:** The PEND DB modifications cannot be reverted from the UI (no op_id exists). DBA-only recovery via manual UPDATE.
- **Severity:** Critical — silent half-applied upload with no audit trail. Confirms prior audit's #6/#7.

#### D-6. [HIGH] DO `qty=0` cancel with only `(rdc, art)` cancels ALL OPEN BDC across all stores
- **file:line:** `pend_alc_service.py:2643, 2687, 2748-2750`.
- **Repro:** CSV has `rdc=R09, article_number=A1, do_qty=0` (no st_cd, no alloc_no).
- **Resulting:** Falls into `cx_global` bucket. The cancel pass for `bucket='global'` runs `UPDATE H SET STATUS='CANCELLED' FROM BDC_HISTORY H JOIN agg ON agg.rdc=H.RDC AND agg.art=H.ARTICLE_NUMBER WHERE H.STATUS='OPEN'`. Every OPEN BDC for `(R09, A1)` across every store gets cancelled.
- **Impact:** Bulk wildcard cancel without any confirmation, scoped only by `(rdc, art)`. A user pasting a placeholder row with empty cells can wipe a day's worth of BDC for the article.
- **Severity:** High — destructive default with no scope confirmation.

#### D-7. [MEDIUM] DO with `allocation_number` mismatching BDC's `(rdc, art)` silently falls through to FIFO
- **file:line:** `pend_alc_service.py:3316-3331`.
- **Repro:** User uploads `(rdc=R09, art=A1, alloc_no=ALLOC123, st_cd=S01, do_qty=5)`, but ALLOC123 was actually issued for `(rdc=R12, art=A1)`.
- **Resulting:** The OUTER APPLY's `WHERE H.ALLOCATION_NUMBER=:alloc AND H.RDC=:rdc AND H.ARTICLE_NUMBER=:art` returns no rows. ST_CD is not overwritten. The row falls into the scoped FIFO pass for `(R09, S01, A1)` if any open PEND exists.
- **Impact:** User intended one BDC, got another. No warning.
- **Severity:** Medium.

#### D-8. [MEDIUM] Concurrent DO uploads for overlapping keys
- **file:line:** `pend_alc_service.py:3192`.
- **Repro:** Two operators upload DOs at the same time, both touching `(R09, S01, A1)`. Each session reads `IS_CLOSED=0` open rows and computes FIFO independently.
- **Resulting:** Both UPDATEs are single set-based statements with row-level locks. SQL Server serializes them, but each session's CTE materialises a snapshot of `open_ranked`. Under RCSI (which is enabled per `prewarm_pend_alc_tables` lifespan), each session sees a consistent snapshot at statement start. If session A applies first and closes the row, session B's CTE still saw it as open and computes apply_qty against the now-stale `need`. The actual UPDATE re-reads via the JOIN and respects new DO_QTY, so the result is bounded by `need` post-A's commit — but session B's `pend_updates` records `qty_added=0` for already-fully-consumed rows. OK at end-state.
- The risk is **`auto_history_closes` race**: if A closes the PEND row (sets IS_CLOSED=1), and B's `apply_do_deductions` sees `was_just_closed=False` because A did it, the `IF any_just_closed` branch in B's run doesn't trigger the auto-close of OPEN BDC history. End state: OPEN BDC history rows remain open even though the underlying PEND is closed. The next `bdc-generate` filter `_NO_OPEN_BDC_PREDICATE` thinks the article is in flight.
- **Severity:** Medium — race condition window is narrow but produces persistent inconsistency.

#### D-9. [LOW] `DO_NUMBER` concat across many uploads exceeds NVARCHAR(100)
- **file:line:** `pend_alc_service.py:3404-3408`.
- **Repro:** Same PEND row receives 20 partial DOs over time, each adding a 10-char DO number.
- **Resulting:** `DO_NUMBER + ', ' + alloc_plan.do_numbers` eventually exceeds 100 chars. SQL Server silently truncates on UPDATE? No — VARCHAR truncation throws SQL error 8152 if not configured to silent-truncate. Will raise.
- **Severity:** Low — eventual operational ceiling, no silent corruption.

#### D-10. [HIGH] Adhoc-closed PEND row whose `IS_CLOSED=1` blocks DO but leaves no trail
- **file:line:** `pend_alc_service.py:3382`.
- **Repro:** Adhoc close ran at T1. PEND for `(R09, S01, A1)` is `IS_CLOSED=1`. SAP DO arrives at T2 for the same key with `do_qty=5`.
- **Resulting:** `apply_do_deductions` filters `WHERE P.IS_CLOSED=0` → no open rows → no apply. `update_bdc_history_with_do`: all OPEN BDC history was cancelled at T1, so no apply either. **`touched=0` on both**.
- **Impact:** SAP shipped 5 units. The data says 0 units shipped. No audit trail in any of the three tables. The next MSA Generate doesn't see the DO either.
- **Severity:** High — physical-vs-data mismatch caused by adhoc-close happening before DO.

## TAB 3 — Adhoc Close (`POST /pend-alc/close-rows` and `/close-rows-file`)

### 3a. How it works today — end to end

**JSON endpoint:** `pend_alc.py:2361-2432`. Body: `CloseRowsRequest` (`pend_alc.py:2356`).
**File endpoint:** `pend_alc.py:2435-2522`. Accepts CSV / XLS / XLSX via `UploadFile`. Parses with `pandas.read_csv` / `read_excel` with `dtype=str`. Required columns: `RDC, ARTICLE_NUMBER`. Optional: `ST_CD, REASON`.

Flow:

1. **Pydantic / pandas decode.**
   - JSON: each row → `{rdc, st_cd, article_number, reason}`. Per-row reason falls back to body-level `reason` falls back to `""`.
   - File: pandas `fillna("")` so NaN cells don't poison SQL match. Column names uppercased + stripped. `article_number = str(...).strip().split(".")[0]` — strips Excel's float-suffix (e.g. `123456.0` → `123456`). Rows with blank `rdc` or `art` are silently skipped.
2. Generate `op_key = uuid.uuid4().hex[:12]`.
3. **`apply_adhoc_close(conn, rows, reason, created_by)`** (`pend_alc_service.py:2878-3008`):
   - Filters: drop rows with blank rdc or art. Each accepted row has `st_cd = stripped str or ""`.
   - Creates `#adhoc_close_<hex>` and bulk-loads via `fast_executemany`.
   - **PEND_ALC close** (`pend_alc_service.py:2949-2968`):
     ```sql
     UPDATE P SET P.IS_CLOSED = 1,
                  P.REMARKS = CASE WHEN P.REMARKS IS NULL OR P.REMARKS='' THEN :prefix
                                    ELSE LEFT(:prefix + ' | ' + P.REMARKS, 500) END
     OUTPUT INSERTED.ID, DELETED.IS_CLOSED, DELETED.REMARKS
     FROM ARS_PEND_ALC P
     JOIN #adhoc_close u
       ON P.RDC = u.rdc
      AND P.ARTICLE_NUMBER = u.art
      AND (u.st_cd = '' OR ISNULL(P.ST_CD,'') = u.st_cd)
     WHERE P.IS_CLOSED = 0
     ```
     `:prefix = f"ADHOC: {safe_reason}"` (`pend_alc_service.py:2917-2918`).
   - **BDC_HISTORY cancel** (`pend_alc_service.py:2973-2987`):
     ```sql
     UPDATE H SET H.STATUS = 'CANCELLED'
     OUTPUT INSERTED.ID, DELETED.STATUS
     FROM ARS_BDC_HISTORY H
     JOIN #adhoc_close u
       ON H.RDC = u.rdc
      AND H.ARTICLE_NUMBER = u.art
      AND (u.st_cd = '' OR ISNULL(H.ST_CD,'') = u.st_cd)
     WHERE H.STATUS = 'OPEN'
     ```
   - Returns `{touched_pend, touched_history, pend_updates, history_updates}`.
4. **`log_operation`** (`pend_alc_service.py:224-247`) with `OP_TYPE='ADHOC_CLOSE'`, `OP_KEY=op_key`, `PAYLOAD={reason, input_keys, pend_updates, history_updates}`.

**Crucial omission:** `apply_adhoc_close` does NOT:
- Call `apply_pend_alc_delta(sign=-1)` to reverse the PEND_QTY effect on MSA / grids.
- Call `bootstrap_msa_pend_sync` / `bootstrap_grid_pend_sync`.
- Update `LAST_DO_AT` / `LAST_BDC_AT` on the closed rows.

So the closed rows still count in `MSA_TOTAL.PEND_QTY` until something else triggers a bootstrap (next MSA Generate, or a separate revert, or a grid build).

**Indexes hit:**
- PEND_ALC: `IX_ARS_PEND_ALC_lookup (RDC, ARTICLE_NUMBER, IS_CLOSED)` — well covered.
- BDC_HISTORY: `IX_ARS_BDC_HISTORY_lookup (RDC, ST_CD, ARTICLE_NUMBER)` + `IX_ARS_BDC_HISTORY_status (STATUS, BDC_DATE)`.

### 3b. Inputs & validation

| Field | Required | Defaulted | Validation |
|---|---|---|---|
| `rdc` | yes | — | Service skips if blank (line 2906). Pydantic accepts empty. |
| `article_number` | yes | — | Same. |
| `st_cd` | optional | `""` (interpreted as "any store") | No validation. |
| `reason` (per-row) | optional | falls back to body `reason` or "" | Truncated to 400 chars (`pend_alc_service.py:2917`). |
| `reason` (body / query) | optional | "" | Same. |

File endpoint additional:
- File must end `.csv` / `.xlsx` / `.xls`.
- `RDC` and `ARTICLE_NUMBER` columns required (case-insensitive after `df.columns.upper()`).
- `pandas.read_csv(..., dtype=str)` — every cell read as string.
- `df.fillna("")` — NaN → empty string. **Critical**: without this, `str(NaN)='nan'` would silently match nothing or worse.
- `art = str(row.get("ARTICLE_NUMBER") or "").strip().split(".")[0]` — drops Excel's auto-floated `.0` suffix. **Doesn't handle scientific notation** (`1.23E+05`).
- Empty file → 400. Unsupported extension → 400.

**Silently accepted that should be rejected:**
- Blank `ST_CD` on a row interpreted as "close every store" (see Edge case A-1 below) — accepted with no scope-confirmation.
- `(RDC, ARTICLE)` not in PEND_ALC at all — silently no-op, `touched_pend=0` reported.
- Article numbers in scientific notation (`1.23E+05` from a corrupted Excel) — not normalised; `str(...).split('.')[0]` returns `1`. Matches nothing usually, but in pathological cases could match an unintended article.
- `reason` with newlines or unicode control chars — accepted verbatim.

| Table | Op | Columns |
|---|---|---|
| `ARS_PEND_ALC` | UPDATE | `IS_CLOSED=1, REMARKS = 'ADHOC: <reason> \| <existing>'` |
| `ARS_BDC_HISTORY` | UPDATE | `STATUS='CANCELLED'` (only rows with STATUS='OPEN') |
| `ARS_PEND_ALC_OPERATIONS` | INSERT | `OP_TYPE='ADHOC_CLOSE'` with pend/history pre-images |

**Revert payload sufficiency** (`pend_alc_service.py:3040-3116`, `_revert_adhoc_close`):
- `pend_updates`: `{pend_alc_id, was_closed (DELETED.IS_CLOSED), old_remarks}` — pre-image is enough to restore IS_CLOSED and REMARKS to exactly what they were.
- `history_updates`: `{history_id, old_status}` — restores STATUS verbatim.
- BUT: there is no record of OTHER changes that happened between the close and the revert. If a fresh `bdc-generate` ran after the close and inserted a NEW BDC_HISTORY row at a new ID, the new row is not in `history_updates` and the revert restores STATUS='OPEN' on the OLD row. - The PEND_QTY effect on MSA (which never got reversed). Revert via `revert_operation` calls `bootstrap_msa_pend_sync` and `bootstrap_grid_pend_sync` at the end (`pend_alc_service.py:1183-1199`), which IS what re-syncs MSA after the revert restores IS_CLOSED=0. So MSA is correct AFTER revert but was wrong DURING the closed window.
- No record of `bdc-generate` runs that happened post-close, so the dual-OPEN-history state (above) is not visible from the ops_log.

### 3d. Edge cases and bugs

#### A-1. [HIGH] Blank `ST_CD` is a wildcard — closes ALL stores for `(RDC, ARTICLE)`
- **file:line:** `pend_alc_service.py:2961, 2981`.
- **Repro:** CSV row `RDC=R09, ARTICLE_NUMBER=A1`, ST_CD column empty (or missing). Open PEND rows exist for `(R09, S01, A1), (R09, S02, A1), (R09, S03, A1)`.
- **Action:** Upload via `/close-rows-file`.
- **Resulting:** All three rows closed. All three stores' OPEN BDC history for that article cancelled.
- **Impact:** User intent was ambiguous; the system picked "destroy everything for this RDC+article". No confirmation prompt.
- **Severity:** High — bulk wildcard destruction by accident.

#### A-2. [HIGH] MSA PEND_QTY stays inflated until next bootstrap
- **file:line:** `pend_alc_service.py:2878` (absence of MSA delta call).
- **Repro:** Close a 50-row session totalling PEND_QTY=500.
- **Starting state:** `MSA_TOTAL.PEND_QTY=500`, `FNL_Q = STK − 500 − HOLD`. Sum of `ARS_PEND_ALC` open rows = 500.
- **Action:** Adhoc close all 50 rows. `IS_CLOSED=1` set; `PEND_QTY` on PEND_ALC computed column becomes... still 500 (because `ALLOC_QTY − DO_QTY` is unchanged — `IS_CLOSED` only filters the rows). The downstream `bootstrap_msa_pend_sync` filters `WHERE IS_CLOSED=0` (`pend_alc_service.py:5130`), so it WOULD correctly produce PEND_QTY=0 for closed rows — but **it is never called from `apply_adhoc_close`**.
- **Resulting:** `MSA_TOTAL.PEND_QTY` stays at 500 even though the 500 units are no longer pending. `FNL_Q` stays deflated. Listing/allocation in the window between close and next MSA Generate under-allocates by 500 units.
- **Impact:** Inverse of M-2's stock leak — this one is conservative (allocates less) but causes operational confusion ("why didn't we allocate this article that has stock?").
- **Severity:** High — confirmed by code path. The audit's prior #4 finding is accurate.

#### A-3. [HIGH] Revert blocker gate uses fields adhoc-close never touches
- **file:line:** `pend_alc_service.py:3011-3037`.

#### A-5. [MEDIUM] Concurrent DO upload during adhoc close — race window
- **file:line:** `pend_alc_service.py:2949` (adhoc close UPDATE) vs `pend_alc_service.py:3399` (DO UPDATE).
- **Repro:** At T0, DO upload starts for `(R09, S01, A1)`. At T0+ε, adhoc close starts for same key.
- **Resulting:** Two UPDATEs against the same PEND row. RCSI gives each its own snapshot; the second to commit wins, but the first commits its OUTPUT capture before the second's filter sees it. If adhoc close commits first (`IS_CLOSED=1`), the DO's CTE `WHERE P.IS_CLOSED=0` filter (evaluated at statement start) saw the row as open and computes apply_qty against it. The UPDATE then re-checks via JOIN but does NOT re-check IS_CLOSED in the WHERE clause — so the DO IS applied to the now-closed row (DO_QTY goes up, IS_CLOSED stays 1 because the `CASE WHEN DO_QTY + apply >= ALLOC THEN 1 ELSE 0 END` only flips to 0 in the false branch, which doesn't actually run — let me re-check).
- Re-reading line 3401: `IS_CLOSED = CASE WHEN P.DO_QTY + alloc_plan.apply_qty >= P.ALLOC_QTY THEN 1 ELSE 0 END`. So if DO partially covers (`< ALLOC_QTY`), this CASE sets `IS_CLOSED=0` — **unconditionally**, even if it was already 1. The DO will FLIP an adhoc-closed row back to open if the DO doesn't fully cover ALLOC_QTY.
- **Severity:** Medium — race window plus an actual data correctness bug (DO UPDATE clobbers `IS_CLOSED` set by adhoc close).
